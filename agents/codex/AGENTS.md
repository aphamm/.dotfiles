These are guiding principles you should follow as a competent engineer.

Write code in the dumbest, most simple way possible. Simple is complicated enough, especially at scale. Do NOT design overcomplicated abstractions that require reasoning about. Everything should be easily traceable in the code. If a simpler approach exists, say so; push back when warranted.

Delete lines of code. Sensible engineers write as little code as possible.

Prefer immutable data by default; reach for mutable contiguous buffers only in measured hot paths. Clear intermediate names. Ask user input for naming conventions. This builds a shared design language with the user. Use one word for one concept, consistently. Do not invent synonyms unless the distinction is real.

Keep data and behavior separate: plain structs and arrays operated on by functions, not objects that bundle state behind hidden dispatch. For a known, closed set of variants, branch on an explicit tag (a `switch`) rather than polymorphism вҖ” every case stays greppable in one place and the compiler can prove exhaustiveness; reach for dynamic dispatch only when the set of types is genuinely open (plugins, third-party extension). When one type dominates memory on a hot path, shrink it and mind layout вҖ” indexes over pointers, struct-of-arrays, sparse data out of band вҖ” but only where it's measured to matter; clarity wins everywhere else.

Fix root causes, not symptoms.

Think before coding. State assumptions, surface tradeoffs, and ask when unclear. 

Prefer a fast, deterministic feedback loop: test, script, curl command, browser check, or replayed fixture. For bugs, prove the user's actual symptom before fixing. For multi-step tasks, give a brief plan with verification checks. 

Tests should verify behavior through public interfaces, not implementation details. Prefer tests that read like specifications. Test one behavior at a time. Prefer deep modules: small interfaces with meaningful behavior behind them. The interface is the test surface.

Record ADRs only for decisions that are hard to reverse, surprising without context, and based on a real tradeoff. Use `CONTEXT.md` as a glossary of shared language with the user.

Write persisted text вҖ” docs, comments, commit bodies, ADRs, `CONTEXT.md` вҖ” to be unsummarizable: so little fluff that removing any words (which is all a summary does) deletes interesting ideas, not filler. Information-theoretically, aim for high entropy per token вҖ” text that is already close to its own shortest encoding, near-incompressible. A summary is lossy compression; when nothing redundant is left, summarizing can only discard signal. Judge this across the whole directory, not one file: state each idea once, in one place, and never restate what another file already carries вҖ” the repo as a whole should be incompressible.

Always prefix commit messages with a random Unicode symbol from: вҳ… вңҺ вҶ— вҳј вҳҫ вңҰ вҢҳ в—Ҳ в–і в–Ҫ в—Ү в–ё в—Ҷ вҹі вҠҳ вҠҷ вӢҲ вҹЎ вҢ— вҶҜ вҮ„ вҮ… вүӢ вҲҝ вҠҝ вӢ® вҠ¶ вҠ· вҠ№ вҹ  вҸҸ вҠҡ вӢұ вҠЎ Never claim AI co-authorship: no `Co-Authored-By: Claude` (or similar) trailer or "generated by" credit.

When pulling documentation always use:

- **Library/API docs**: `curl "https://context7.com/<github-owner>/<github-repo>/llms.txt?tokens=10000"`
- **Repo architecture**: `WebFetch("https://deepwiki.com/<github-owner>/<github-repo>")`
- **Research papers**: `curl "https://arxiv2md.org/api/json?url=<arxiv-doi>"`
- **YouTube transcripts**: `yt-dlp --write-auto-sub --sub-lang en --skip-download --sub-format vtt -o - "<url>"`

For Python, always use `uv`. For TypeScript, always use `pnpm` and `pnpm dlx` for one off CLI tools.
